# MCP server Demo + 幻觉对比

> 「可溯源中国历史故事」MCP server 的一键 demo，与「裸 LLM 凭记忆 vs 查本 server」的
> 逐问对比。配置与诚实边界见 [README](../README.md)。
>
> **一句话卖点**：裸 LLM 会自信地**编造段级出处**、把**语料外的人**塞进答案、把**古今地名**张冠李戴；
> 本 server 只返回语料里真实存在的内容，每条带【书→篇→段】，机器判断/生成物如实标 `review_status`。

---

## 1. 一键跑 demo（可截图/录屏）

```bash
PYTHONPATH=src python3 scripts/mcp_demo.py --db /path/to/corpus.db   # 拉起 server, 走真实 stdio JSON-RPC 四场景
```

覆盖：① 今地名反查（西安→消歧→南阳）② 品质代表故事/人物（忠）③ 人物画像 + 他者评价（曹操）
④ 跨书融合事件 + 逐源出处（官渡之战）。`mcp_demo.py` 本身也是一份**最小 MCP 客户端参考实现**。

手测一次握手见 [README](../README.md) 的 "Try one handshake by hand"。

## 2. 幻觉对比（同一问题：裸 LLM vs 查本 server）

> 「裸 LLM」列＝无检索、凭训练记忆作答时的**典型失败模式**（可自行复现，见 §3）。
> 「本 server」列＝上面 demo 的**真实返回**（先秦-汉魏 9 部书语料，逐条可溯源）。

### Q1　「史书里别人怎么评价曹操？原话和出处。」

| | 输出 | 问题 |
|---|---|---|
| **裸 LLM** | 「许劭评曹操『治世之能臣，乱世之奸雄』，见《三国志·武帝纪》」 | 名句真伪掺杂、**出处张冠李戴**（该语出裴注引《异同杂语》/孙盛，非《武帝纪》正文）；无法给段级定位；读者无从核验 |
| **本 server**<br>`get_person("曹操")` | 逐条他者评价 + 出处：<br>· 〔资治通鉴·汉纪五十五 段 9〕「曹公有雄才远畧，决机无疑，法一而兵精，能用度外之人，所任各尽其力…」<br>· 〔资治通鉴·汉纪五十六 段 24〕「曹公允恭明哲，翼戴天子，伐叛柔服，宁静四海」<br>画像另标 `review_status=draft`（LLM 综合、未逐条人核） | 每条是**原文照抄 + 段级出处**，可翻到那一段核对；画像明确标机器生成 |

### Q2　「史料里体现『忠』的代表事件/人物，要原文证据。」

| | 输出 | 问题 |
|---|---|---|
| **裸 LLM** | 「关羽、岳飞、文天祥……『人生自古谁无死，留取丹心照汗青』」 | 举的人多**在本语料时代之外**（先秦-汉魏）；名句无法定位到具体书篇段；把"名气"当"证据" |
| **本 server**<br>`query_by_quality("忠")` | · [事件] 鞍之战及战后齐晋交涉 — 逄丑父与齐顷公交换位置、代君被俘 · 强度 typical · `auto_approved`<br>　证据〔史记·齐太公世家第二 段 12〕<br>· [人物] 公孙鞅 …〔史记·范雎蔡泽列传第十九 段 19–25〕「极身无贰虑，尽公而不顾私」 | 事件/人物均**限定在语料内**、带 `evidence_quote` 真子串 + 段级出处；`review_status` 标机审档 |

### Q3　「发生在今河南南阳的古籍故事有哪些？出处。」

| | 输出 | 问题 |
|---|---|---|
| **裸 LLM** | 「南阳是诸葛亮躬耕地，《出师表》『臣本布衣，躬耕于南阳』……」 | 拿最出名的联想充数、卷入"躬耕地"千年公案而不自知；**给不出古名→今地的映射依据**，更无段级出处 |
| **本 server**<br>`query_by_place("南阳")` | 今地 河南南阳，共 562 个故事，例：<br>· 穆公求贤得百里傒〔史记·秦本纪第五 段 18–19〕原文作『宛』· 映射 `auto_approved`/存疑<br>· 始皇即位与早期征伐〔史记·秦始皇本纪第六 段 1–2〕原文作『宛』 | **今地名反查**：从原文『宛』确定性反查回段；映射档 + 存疑标注如实透出；story 标机器分割 |

### Q4　「鸿门宴这件事，哪些史书记载？各自哪一段？」

| | 输出 | 问题 |
|---|---|---|
| **裸 LLM** | 「《史记·项羽本纪》和《汉书·高帝纪》，大约在……第X段」 | **段号几乎必为编造**（模型无稳定段级坐标）；跨书对读要靠人重新翻书；无法保证真有那段 |
| **本 server**<br>`search_events("鸿门宴")` | 跨书合并事件，逐源：<br>· 〔主叙〕资治通鉴·汉纪一 段 4–8<br>· 〔详述〕史记·项羽本纪第七 段 12–16<br>· 〔旁证〕史记·留侯世家第二十五 段 4<br>`canonical_summary` 标 machine-generated；`review_status=approved`（机审批量过审的可信推定） | 同一史事跨书记载**已对齐到段**、逐源标角色（主叙/详述/旁证）；融合叙述明标机器生成 |

## 3. 自己复现

1. **裸 LLM**：把上面任一问题原样问任意大模型（**不挂**本 MCP server），看它能否给出**可核验的段级出处**、是否把语料外的人或错误出处塞进答案。
2. **查本 server**：在 Claude Desktop / Cline 里挂上本 server（见 [README](../README.md) 配置），问同样的问题，看返回的【书→篇→段】能否逐条翻回原文核对。
3. 对读差异：**「带出处、限定语料内、机器判断如实标注」 vs 「流畅但不可核验」**。

## 4. 本 server 不宣称什么（诚实边界）

- **不宣称"逐条人工核校"**：事件 `approved` 多为机审批量过审的**可信推定**；人物画像与部分品质映射为 `draft`；这些都在返回的 `review_status` / `*_note` 字段里如实标注。
- **不宣称译文/标点为人工**：原文是公版白文 + **机器标点/分段**，白话是**全库机译**（machine-generated），随返回透出（AIGC 标识合规）。
- **不消灭下游幻觉**：本 server 提供**可溯源的检索事实**；挂它的下游 LLM 仍可能围绕检索结果二次脑补——出处是给**人复核**用的锚，不是"端到端无幻觉"的保证。
- **不覆盖全部历史**：语料仅**先秦-汉魏 9 部**（史记/汉书/后汉书/三国志/左传/论语/孟子/吕氏春秋/资治通鉴）；查不到 ≠ 史无此事，只是不在本库。
